# Prompt Workbench 解释手册

这份手册偏“理解项目”，不是教你点哪里，而是说明这个工作台为什么这样设计、每个文件负责什么。

## 1. 这个页面的定位

`PromptWorkbench` 不是一个通用聊天机器人，而是一个“提示词生成工作台”。

它解决的核心问题是：

- 日常和 AI 对话时，不知道怎么组织一句高质量提问
- 同一类任务反复出现，但每次都要重新写提示词
- 提示词会写，但不想每次都从空白开始搭结构

所以它没有直接接模型 API，而是先把“提问结构”标准化，再让你复制给任意 AI 使用。

## 2. 为什么做成单文件网页

当前版本把主要界面和逻辑集中在：

- [PromptWorkbench.html](/D:/study/提示词助手/langgpt-main/PromptWorkbench.html)

这样做有几个好处：

- 不依赖前端打包工具
- 不需要安装额外依赖
- 复制、分发和调试都简单
- 对个人使用场景足够直接

这也是为什么当前项目很适合作为“个人提示词工作台”的第一版。

## 3. 页面结构说明

### 模板层

页面里内置了一组模板对象，每个模板包含：

- `id`
- `group`
- `title`
- `summary`
- `fields`
- `prompt`

这里的思路是“参数化提示词”。

也就是说，用户不需要每次自己重写整段提示词，只需要填写变量，再由程序把变量代入模板。

### 状态层

页面会记录：

- 当前选中的模板
- 搜索条件
- 字段填写内容
- 语气、长度、输出方式
- 收藏项
- 补充背景

这些状态存放在浏览器本地存储中，因此刷新页面后通常还能保留上次填写内容。

### 生成层

生成逻辑的本质是三步：

1. 根据当前模板拿到字段定义
2. 用用户填写的数据替换模板里的占位符
3. 把额外偏好和补充背景拼接到结果末尾

所以右侧看到的内容，不是固定文本，而是模板和输入内容组合后的结果。

## 4. 为什么分成左中右三块

这是为了把“找模板”“填信息”“拿结果”三个动作拆开。

### 左侧

解决“选哪种模板”的问题。

### 中间

解决“这次具体要什么”的问题。

### 右侧

解决“最后给 AI 发什么”的问题。

这种结构适合高频重复使用，因为它让用户每次只关注一个步骤，不需要在一页上同时处理所有信息。

## 5. 为什么有语气、长度、输出方式

很多时候，用户真正需要的不只是“内容”，还需要“内容的呈现方式”。

比如同一个问题：

- 可以要求简洁
- 可以要求详细
- 可以要求先结论后解释
- 可以要求表格输出

这些不是任务本身，但会显著影响 AI 的输出质量。所以这里把它们抽成了统一控制项，而不是让用户每次手写。

## 6. 为什么还保留 Markdown 模板文件

仓库里除了网页，还有：

- [Prompts/日常AI对话模板.md](/D:/study/提示词助手/langgpt-main/Prompts/日常AI对话模板.md)

保留这类 Markdown 文件有两个作用：

- 方便阅读和整理模板原文
- 方便后续继续扩展成模板库、知识库或文档站

换句话说，网页偏“使用界面”，Markdown 偏“内容底稿”。

## 7. 本地服务为什么这么简单

服务端文件是：

- [server.js](/D:/study/提示词助手/langgpt-main/server.js)

它的作用不是做复杂后端，而只是：

- 把根路径 `/` 映射到 `PromptWorkbench.html`
- 提供最基础的静态文件访问
- 让浏览器通过 `18888` 端口稳定访问页面

因为当前项目没有数据库、没有账号系统、没有 API 调用，所以用轻量静态服务就够了。

## 8. 这个项目现在还不做什么

当前版本故意没有做这些：

- 不直接调用大模型 API
- 不做登录系统
- 不做多人协作
- 不做云端同步
- 不做复杂模板编辑器

原因不是不能做，而是当前目标是先把“个人高频使用”这件事跑顺。

## 9. 后续适合怎么扩展

如果后面继续往前走，比较自然的方向有：

- 增加“自定义模板”
- 增加“历史记录”
- 增加“导入/导出模板”
- 增加“按行业分类模板”
- 增加“直接调用 AI API”
- 把单文件网页拆成正式前端项目

如果只是个人使用，当前形态已经够轻。

如果后面想做成长期维护的产品，再考虑引入更完整的前后端结构会更合适。
